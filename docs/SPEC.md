# 仕様書（詳細）

このドキュメントでは、超マイスケジュール 2026 ジェネレータの機能要件を細かく定義します。具体的な挙動・データ構造・API などを記載し、実装やテストがしやすい形にまとめます。

---

## 1. 用語定義

- **セッション**: ユーザーが登録する予定（イベント）。
- **タイムテーブル**: セッションを時間軸で表示した画面。
- **画像出力**: 作成中のタイムテーブルを PNG 形式でエクスポートする機能。
- **カレンダー出力**: 予定を `.ics` 形式でエクスポートし、他のカレンダーに読み込める形式にする機能。

---

## 2. ユーザー操作フロー（基本）

1. 画面を開く
2. セッションを追加
3. タイムテーブルに表示される
4. 必要に応じて編集・削除
5. 画像を出力 or カレンダー形式でダウンロード

---

## 2.1 データ保存とユーザー識別

- **アカウント作成やログインは想定しない**（ユーザーは匿名利用）
- セッションデータは **ブラウザのストレージ（localStorage / IndexedDB / cookie など）** に保存する
  - サーバーが存在する場合でも、状態はクライアント毎に分離される
  - 別ブラウザ・別デバイス間の同期は行わない
- **時間は JST で管理**し、入力/表示ともに日本時間基準で扱う

---

## 3. セッション（データモデル）

### 3.1 フィールド

| 項目 | 型 | 必須 | 備考 |
|---|---|---|---|
| id | UUID（文字列） | ✅ | 自動生成 | 
| title | 文字列 | ✅ | 1〜100 文字程度 | 
| startAt | ISO 8601 datetime (JST) | ✅ | 例: `2026-04-25T10:00:00+09:00` |
| endAt | ISO 8601 datetime (JST) | ✅ | startAt より後 | 
| venue | 文字列 | ✅ | 会場 / ステージ名 | 
| tags | 文字列配列 | 任意 | カンマ区切り or タグボタン | 
| memo | 文字列 | 任意 | セッション備考 | 
| color | 文字列 | 任意 | セッション枠の背景色 (16 進色) |
| createdAt | ISO 8601 datetime | ✅ | 自動生成 | 
| updatedAt | ISO 8601 datetime | ✅ | 自動更新 | 
| day | 文字列 (`2026-04-25` など) | ✅ | 画面切り替えやエクスポートの絞り込みに使用 |

### 3.2 バリデーションルール

- `title` は空文字禁止（trim 後も 1 文字以上）
- `startAt < endAt` を満たす必要がある
- `startAt` / `endAt` は超会議 2026 の期間内（2026-04-25 00:00 〜 2026-04-26 23:59）であることを推奨（ただし柔軟性として多少の前後は許可）
- `venue` は空文字禁止
- `tags` は最大 10 個程度、各タグは 1〜30 文字程度を推奨
- `color` は 16 進カラー形式（例: `#FF0080`）

---

## 4. タイムテーブル表示

### 4.1 画面の構成

- **日付切り替え**: 1 日目 / 2 日目で切り替え可能
- **時間軸**: 05:00〜26:00 などの連続表示（必要に応じて上下スクロール）
- **会場ごとの列**: `venue` ごとに列を分ける（会場名を列ヘッダに表示）
- **セッションブロック**: `startAt`〜`endAt` に応じた高さで表示
- **重複検出**: 同じ会場内で時間が重なるセッションがあった場合、以下のいずれか（または両方）を行う
  - 背景色を変える / 目立つ枠線を追加
  - エラーメッセージや警告アイコンを表示

### 4.2 表示要件

- セッションブロックに `title` を表示（幅が足りない場合は省略）
- マウスオーバー/タップで `memo` や `tags` など詳細をツールチップ/ポップアップで表示
- ブロッククリックで編集モーダルを開く
- 日付切り替えは画面遷移せずにクライアント側で切り替える
- **モバイル端末でも操作しやすいレスポンシブUIを必須とする**（モバイルファースト、タッチ操作対応）

---

## 5. 画像出力（PNG）

### 5.1 目的

- スマホや SNS で共有しやすい横長フォーマットを生成する
- 超会議 2026 の世界観（公式配色／フォント感）を想起させる

### 5.2 仕様

- **フォーマット**: PNG
- **サイズ**: 以下のプリセットを用意
  - **横長（X/Twitter 向け）**: 1920x1080（または 1600x900）
  - **縦長（Instagram フィード／ストーリー向け）**: 1080x1350（フィード）または 1080x1920（ストーリー）
- **出力に含める情報**:
  - タイトル（例: 「超マイスケジュール 2026」）
  - 日付（1 日目 / 2 日目/両日）
  - セッションのタイムテーブル（時間軸 + 会場列）
  - 重要なセッションは強調表示（例: 色・枠）
  - 生成日時 / 作成者名（任意）

### 5.3 UI 操作

- 画像出力ボタンを押すと 1) 出力対象日（1 日目 / 2 日目 / 全日） 2) フォーマット（横長/縦長） 3) 解像度（標準/高解像度） 4) 対象ページ（全体/1 日目/2 日目）などを選べるオプションを表示
- 処理中はローディング表示を出す
- 出力完了後、自動ダウンロードまたは外部リンクで保存

### 5.4 実装ヒント

- **クライアントサイドでの生成を想定**: HTML/CSS を Canvas にレンダリングして画像化する（`html-to-image`、`dom-to-image`、`html2canvas` など）
- サーバーサイドで生成する場合は、クライアント側生成の代替として `sharp` + `node-canvas` などを使うこともできるが、基本はクライアント上で完結させる

---

## 6. カレンダー出力（.ics）

### 6.1 目的

- 作成した予定を一般的なカレンダーアプリに読み込める形式でエクスポートする

### 6.2 対象

- Google カレンダー（.ics インポート）
- Apple 標準カレンダー（.ics）

### 6.3 仕様

- **形式**: iCalendar（RFC 5545）
- **タイムゾーン**: 日本時間（`Asia/Tokyo`）
- **各イベントのフィールド**:
  - `UID`: セッション ID
  - `DTSTART` / `DTEND`: `startAt` / `endAt`
  - `SUMMARY`: `title`
  - `LOCATION`: `venue`
  - `DESCRIPTION`: `memo` + `tags` などを改行で追加
  - `DTSTAMP`: 生成日時
- **エクスポートの範囲**:
  - 選択中の日（1 日目 / 2 日目）
  - 全日
  - 任意の期間（開始日・終了日）

### 6.4 UI 操作

- カレンダー出力メニューから期間を選択してダウンロード
- 生成した `.ics` は直接ダウンロード（`Content-Disposition: attachment`）する

---

## 7. API（バックエンド）

- このアプリは **アカウントレスでクライアント側ストレージに状態を保存する** 想定のため、必ずしもバックエンド API が必要ではありません。
- ただし、サーバーを用意する場合は以下のエンドポイントを提供することで、ブラウザから簡単にデータを永続化できます。

### 7.1 エンドポイント

#### セッション CRUD

| メソッド | パス | 説明 |
|---|---|---|
| GET | `/api/sessions` | 全セッションを取得（クエリで日付や期間を絞り込み） |
| GET | `/api/sessions/:id` | 指定 ID のセッションを取得 |
| POST | `/api/sessions` | セッションを追加 |
| PUT | `/api/sessions/:id` | セッションを更新 |
| DELETE | `/api/sessions/:id` | セッションを削除 |

#### エクスポート

| メソッド | パス | 説明 |
|---|---|---|
| GET | `/api/export/png` | （クライアント側生成が基本のためオプション）タイムテーブルを PNG で出力 |
| GET | `/api/export/ics` | セッション一覧を ICS で出力 |

### 7.2 リクエスト/レスポンス例

#### POST `/api/sessions`

```json
{
  "title": "基調講演",
  "startAt": "2026-04-25T10:00:00+09:00",
  "endAt": "2026-04-25T11:00:00+09:00",
  "venue": "メインステージ",
  "tags": ["講演", "注目"],
  "memo": "開会式を兼ねる",
  "color": "#FF0055"
}
```

成功時（201）

```json
{
  "id": "uuid",
  "title": "基調講演",
  "startAt": "2026-04-25T10:00:00+09:00",
  "endAt": "2026-04-25T11:00:00+09:00",
  "venue": "メインステージ",
  "tags": ["講演", "注目"],
  "memo": "開会式を兼ねる",
  "color": "#FF0055",
  "createdAt": "...",
  "updatedAt": "..."
}
```

エラー時（400）

```json
{ "error": "startAt must be before endAt" }
```

## 8. （任意）リアルタイム同期（socket.io）

- アカウント不要・クライアント保存が前提のため、必須ではない
- 必要に応じて、同一ブラウザの別タブ間で状態を共有したい場合は実装を検討する
- サーバーは変更イベント（`session:created`, `session:updated`, `session:deleted`）を emit
- クライアントはイベントを受け取り、ローカル状態を更新

---

## 9. テスト要件

- API の単体テスト（Jest + supertest）
  - 正常系: CRUD + エクスポート
  - 異常系: バリデーションエラー、存在しない ID へのアクセス
- **クライアントストレージ（localStorage/IndexedDB）への保存・読み込み** のテスト
- 画像出力・ICS 出力はスモークテスト（返却ステータス・Content-Type など）
- フロントエンド（あれば）の単体テストは最低限の UI ロジック検証

---

## 10. 運用・その他

- `.env` で DB 接続情報やポートを管理
- マイグレーションは `knex` で管理し、`npm run migrate:latest` で反映
- ログは `winston` で出力し、必要に応じてファイル/コンソールを切り替え
